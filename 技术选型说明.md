# 技术选型说明

> 项目：对话翻译桌面应用 (Dialogue Translation Application)  
> 版本：v1.0  
> 日期：2026-04-14

---

## 一、跨平台桌面框架：Electron + React

### 选择理由

1. **生态成熟**：Electron 是目前最成熟的跨平台桌面框架，VS Code、Slack 等产品验证了其工程可靠性。npm 生态中的海量包（`better-sqlite3`、`tinyld` 等）可直接使用，无需重新发明轮子。

2. **Node.js 原生模块支持**：本项目需要调用 C++ 编译的 Whisper.cpp 和 Python 的 CTranslate2。Electron 主进程原生支持 `child_process`、`worker_threads`，与这些外部引擎集成非常自然。

3. **React + TypeScript**：React 19 + TypeScript 6 组合提供了强类型保证和组件化抽象，配合 Zustand 轻量状态管理，开发效率极高。Vite 8 的 HMR 在开发阶段几乎是即时响应。

4. **安全隔离模型**：Electron 的 `contextBridge` 机制确保渲染进程无法直接访问 Node.js API，所有能力通过白名单式 IPC 暴露，降低了安全风险。

5. **调试体验**：Chrome DevTools 可直接调试渲染进程，开发效率高。

6. **Native 能力完善**：系统托盘、全局快捷键、文件系统、通知等 API 开箱即用。

### 代价与取舍

- 安装包基础体积 ~80-120MB（Chromium 内核），但对于需要下载 ~150MB Whisper 模型的应用来说，框架体积不是瓶颈。
- 内存基础占用 ~150MB，在桌面设备上可接受。
- 需注意 Main/Renderer 进程隔离，计算密集型任务必须放在 Main 或子进程中。

### 考虑过的替代方案

| 替代方案            | 优势                                | 放弃理由                                                             |
| ------------------- | ----------------------------------- | -------------------------------------------------------------------- |
| **Tauri**           | 体积更小（~5MB），Rust 后端性能优秀 | Rust 后端与 Python/C++ 外部进程的集成不如 Node.js 直观，生态相对年轻 |
| **Flutter Desktop** | UI 层优秀，跨平台一致性好           | 与 Native 进程交互需要额外的 FFI 层，对 ML 推理场景增加了复杂度      |

### 技术栈版本

| 组件             | 版本    | 说明                                     |
| ---------------- | ------- | ---------------------------------------- |
| Electron         | >= 41.x | Chromium 124+, Node.js 20+               |
| React            | 19.x    | + TypeScript 6                           |
| Vite             | 8.x     | 渲染层开发与构建由 Vite 驱动             |
| Tailwind CSS     | 4.x     | 与 Vite 插件集成，维持原子化样式开发体验 |
| electron-builder | 26.x    | 同时输出 .dmg / .exe                     |

## 二、语音识别：Whisper.cpp

### 选择理由

1. **多语言一体**：一个 base 模型（~148MB）覆盖 99 种语言的识别 + 语种检测，不需要为每种语言单独部署模型。
2. **纯 C++ 实现**：无 Python 依赖，既可走系统 PATH 中的 whisper 可执行文件，也可在生产包中直接内置 `whisper-server` / `whisper-cli`。
3. **社区活跃**：Homebrew 可一键安装，持续有性能优化和新模型发布。
4. **支持量化模型**：q5_0 / q8_0 量化版本可大幅减小体积和推理时间。
5. **可选 GPU 加速**：支持 Apple Metal / CUDA 加速（可选），为未来优化留有空间。

### 集成方式选择：`whisper-server` 常驻优先，CLI 兜底

当前实现优先启动常驻 `whisper-server`，通过本地 HTTP 接口完成推理；如果服务端二进制不可用，再回退到 `whisper-cli` 单次调用。这比早期纯 CLI 方案更适合流式识别，因为模型可以常驻内存，并在应用启动后异步预热，降低首段识别延迟。

| 对比维度          | `whisper-server` + CLI   | N-API Addon                 |
| ----------------- | ------------------------ | --------------------------- |
| Electron 版本兼容 | 不受 Node ABI 约束       | 受 Node / Electron 版本限制 |
| 进程隔离          | 强，崩溃可自动重启或回退 | 弱，原生崩溃影响更直接      |
| 首次识别体验      | 预热后更低延迟           | 低                          |
| 打包复杂度        | 通过脚本统一构建二进制   | 需要维护原生绑定            |
| 选择              | ✅ 采用                  | ❌ 放弃                     |

当前打包流程会直接从源码构建 `whisper-server` / `whisper-cli`，macOS 构建默认启用 Metal；开发模式则兼容系统 PATH 中的 whisper 可执行文件。

### 模型选择

| 模型                 | 大小   | 适用场景                    |
| -------------------- | ------ | --------------------------- |
| `ggml-base.bin`      | ~148MB | ✅ 默认推荐，平衡速度与精度 |
| `ggml-small.bin`     | ~488MB | 更高精度，但推理更慢        |
| `ggml-base-q5_0.bin` | ~更小  | 量化版本，速度更快          |

### 音频格式要求

- 采样率：16kHz
- 声道：mono
- 格式：float32 PCM

前端通过 Web Audio API 录音后，使用 `OfflineAudioContext` 重采样到 16kHz，编码为 WAV 格式传递给 Whisper。

## 三、翻译引擎：Opus-MT + CTranslate2

### 选择理由

1. **Opus-MT 模型覆盖广**：Helsinki-NLP 在 HuggingFace 发布了数百个语言对模型，通过 en 中转可覆盖任意两种语言间的翻译。
2. **CTranslate2 推理优化**：专为 Transformer 翻译模型设计的 C++ 推理引擎，INT8 量化后 CPU 推理速度极快（单句 < 1s）。
3. **按需下载**：用户只需下载自己用到的语言对模型（每对 ~50-100MB），而非一个巨型模型。

### 集成方式选择：JSON Line 微服务，开发态 Python / 生产态冻结二进制

当前实现仍然采用 stdin/stdout JSON Line 协议，但把“协议”与“运行入口”分离了：开发环境直接运行 `translate_server.py`，生产环境则运行 PyInstaller 冻结后的 `translate-server` 二进制。这样既保留了 Python 生态对 CTranslate2 的成熟支持，又降低了终端用户的环境依赖。

| 对比维度          | JSON Line 子进程            | HTTP 微服务      | Node.js 直接调用       |
| ----------------- | --------------------------- | ---------------- | ---------------------- |
| 复杂度            | 低（无端口管理）            | 中（需管理端口） | 高（binding 不成熟）   |
| 开发 / 生产一致性 | 高（协议一致，入口不同）    | 中               | 低                     |
| 额外开销          | 无网络栈开销                | 网络栈 + 序列化  | 无                     |
| CTranslate2 支持  | ✅ 官方 Python API 主力维护 | ✅ 可行          | ❌ Node binding 不成熟 |
| 选择              | ✅ 采用                     | ❌ 放弃          | ❌ 放弃                |

### Pivot 中转策略

所有非英语间翻译自动通过英语中转（如 zh → en → fr），理由：

1. **英语相关模型质量最高**：Opus-MT 以英语为中心训练，en↔X 模型质量远高于 X↔Y 模型。
2. **降低模型管理复杂度**：用户无需管理 O(n²) 的语言对，只需 X↔en 即可。
3. **代价可控**：Pivot 翻译延迟约为直接翻译的 2 倍（~600-1500ms），仍在可接受范围。

## 五、总结

本项目的技术选型遵循以下原则：

1. **本地优先**：所有 AI 推理在设备本地执行，不依赖云端 API
2. **集成简单**：优先选择与 Node.js 生态集成最自然的方案（常驻二进制、JSON Line 管道）
3. **按需加载**：模型不打包在安装包中，用户按需下载
4. **代价透明**：每个技术决策的代价和取舍都明确记录
5. **生产自包含**：安装包尽量内置推理运行时，减少终端用户额外配置

这些选择共同构成了一个体积可控、集成简洁、完全离线的桌面翻译应用技术栈。
