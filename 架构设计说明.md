### 1.2 进程模型

| 进程                             | 职责                                        | 生命周期                       |
| -------------------------------- | ------------------------------------------- | ------------------------------ |
| **Renderer**                     | UI 渲染、用户交互、音频采集 (Web Audio API) | 与窗口同生命周期               |
| **Main**                         | IPC 路由、服务编排、文件系统、窗口管理      | 应用生命周期                   |
| **翻译服务进程**                 | CTranslate2 翻译推理                        | 常驻，按需启动，异常自动重启   |
| **whisper-server / whisper-cli** | Whisper 语音识别                            | 常驻服务优先，CLI 作为兜底路径 |

**设计原则**：

- ML 推理绝不在主进程中阻塞执行
- 渲染进程只做 UI，不直接访问任何系统资源
- 所有跨进程通信走 IPC，类型安全

## 二、核心数据流

### 2.1 键盘输入 → 翻译

```
用户输入 → tinyld 语种检测 → IPC saveMessage → SQLite 存储
                                     ↓
                              翻译开关开启?
                          ┌── 是 → 检查模型 → 可用 → Python 推理 → 更新 UI + DB
                          └── 否 → 仅显示原文
```

### 2.2 语音输入 → 流式识别 → 翻译

```
麦克风权限检查 → AudioWorklet → Ring Buffer → VAD 检测
                ↓ (检测到说话)
         增量缓冲 → 静音阈值触发分段 → 送入 Main
                ↓
     whisper-server（优先） / whisper-cli（回退）
                ↓
      文本 + 语种 → 草稿/稳定双层状态 → 实时 UI 更新
                ↓
      稳定片段进入流式翻译缓冲 → 录音结束后保存完整消息
```

### 2.3 流式翻译

```
长文本 → 按句号/问号/感叹号分句
    → 句子1 → Python 翻译 → 立即显示译文1
    → 句子2 → Python 翻译 → 追加译文2
    → ...
    → 完成 → 更新数据库
```

## 三、IPC 通信协议

所有 IPC 通道定义在 `src/shared/ipc-channels.ts`，采用命名空间式组织：

```
APP:          get-platform, get-microphone-access-status, request-microphone-access
STORAGE:      get-messages, save-message, update-translation, clear-messages,
              delete-message, export-messages
TRANSLATE:    text, text-stream, text-stream-chunk, detect-language,
              check-model, check-installed, install-model, install-pair,
              delete-pair, all-model-status, install-progress
WHISPER:      transcribe, stream-segment, check-availability,
              stream-start, stream-audio, stream-end, stream-partial, stream-commit
AUDIO:        start-recording, stop-recording, status, toggle
MODEL:        download, status, download-progress
PERFORMANCE:  get-report, take-snapshot
```

- `invoke/handle` 用于请求-响应模式
- `send/on` 用于单向事件推送（进度、流式数据）

## 四、安全设计

| 安全措施                 | 说明                                           |
| ------------------------ | ---------------------------------------------- |
| `contextIsolation: true` | 渲染进程与 Node.js 完全隔离                    |
| `nodeIntegration: false` | 渲染进程不能直接 require Node 模块             |
| contextBridge 白名单     | 仅暴露明确定义的 IPC API                       |
| Zod Schema 校验          | 关键 IPC 入参在 Main 进程统一校验              |
| Rate Limiter             | 翻译与存储写入限流，降低滥用和意外刷写风险     |
| 输入校验                 | 消息长度限制 (MAX_MESSAGE_LENGTH = 5000)       |
| Buffer 限幅              | 流式音频上传前限制最大采样数，避免内存滥用     |
| SHA256 校验              | Whisper 模型下载与翻译模型 manifest 完整性校验 |
| 无远程通信               | 所有 AI 推理本地执行，用户数据不离开设备       |

## 六、AI 使用方法与总结

### 6.1 开发过程中的 AI 使用

整个项目开发过程中，在以下环节使用了 AI 工具辅助：

| 环节                | 使用的 AI 工具 | 具体作用                                         |
| ------------------- | -------------- | ------------------------------------------------ |
| **需求分析**        | GitHub Copilot | 梳理 PRD 需求，拆解功能模块                      |
| **架构设计**        | GitHub Copilot | 生成系统架构图、模块划分建议、进程模型设计       |
| **代码编写**        | GitHub Copilot | 组件实现、服务编写、IPC 通道注册、类型定义       |
| **Python 翻译服务** | GitHub Copilot | translate_server.py 协议设计、翻译结果后处理逻辑 |
| **流式功能实现**    | GitHub Copilot | VAD 语音检测、分句翻译、实时 UI 更新、双层状态机 |
| **音频处理**        | GitHub Copilot | WAV 编码、OfflineAudioContext 重采样到 16kHz     |
| **调试与优化**      | GitHub Copilot | 翻译结果后处理（去重复、截断）、错误定位         |
| **模型转换脚本**    | GitHub Copilot | convert-opus-mt.py 脚本、模型格式转换逻辑        |
| **技术文档编写**    | GitHub Copilot | 技术选型文档、架构设计文档、性能分析报告         |
| **单元测试**        | GitHub Copilot | 测试用例生成、mock 设计                          |

### 6.2 AI 价值最高的场景

1. **跨语言集成**：Node.js ↔ Python 子进程通信的 JSON Line 协议设计。AI 帮助快速生成了双端一致的序列化/反序列化逻辑，避免了手工对齐协议字段的繁琐。

2. **音频处理代码**：WAV 编码（encodeWAV）、OfflineAudioContext 重采样到 16kHz 等音频处理代码繁琐但标准化，AI 生成效率远高于手写，且不容易出现字节偏移错误。

3. **类型定义**：TypeScript 类型系统贯穿 Renderer → Preload → Main 三层。AI 帮助保持类型一致，减少了跨层类型不匹配的 bug。

4. **翻译后处理**：CTranslate2 的输出有时包含重复片段或不完整句子，AI 帮助设计了 `_postprocess` 函数中的正则清理规则。

### 6.3 AI 需要人工修正的场景

1. **whisper-cpp 输出解析**：AI 初始生成的 stderr 解析逻辑不完全匹配实际输出格式。whisper-cpp 的输出在不同版本间有差异，需要根据真实运行结果逐步调整正则表达式。

2. **Opus-MT 模型 ID**：不同语言对的 HuggingFace 模型 ID 命名不一致（如 en→ja 是 `opus-mt-en-jap` 而非 `opus-mt-en-ja`，ko 使用 `opus-mt-tc-big-*` 而非 `opus-mt-*`）。AI 生成的初始映射表有多处错误，需要逐个在 HuggingFace 上核实。

3. **tinyld CJK 检测**：`tinyld` 对中日韩文本的检测不够准确（日文常被误判为中文），最终补充了 Unicode 范围正则作为前置判断，这是 AI 未能预见到的边界情况。

4. **Electron 安全配置**：AI 生成的 preload 脚本初始版本直接暴露了 `ipcRenderer.on`，存在安全隐患。需要人工审查并改为包装后的安全 API。

### 6.4 改进 AI 使用的反思

| 环节               | 当前做法                                     | 可改进方向                                                       |
| ------------------ | -------------------------------------------- | ---------------------------------------------------------------- |
| **模型兼容性验证** | AI 生成代码 → 手动运行验证 → 发现问题 → 修正 | 可先让 AI 生成针对具体模型的集成测试，提前发现兼容性问题         |
| **跨进程协议设计** | AI 一次生成完整协议 → 联调时发现不匹配       | 可先让 AI 生成协议 Schema + 双端验证代码，确保协议对齐           |
| **音频格式处理**   | AI 生成转换代码 → 运行时发现采样率不对       | 可先让 AI 生成音频格式的单元测试（验证采样率、声道数、编码格式） |
| **文档编写**       | 手动整理后让 AI 润色                         | 可在开发过程中让 AI 持续记录决策日志，最终自动生成文档           |
| **错误处理**       | 功能完成后补充错误处理                       | 可在设计阶段让 AI 列出每个模块的可能失败场景，前置错误处理设计   |
