### 1.2 进程模型

| 进程                             | 职责                                        | 生命周期                       |
| -------------------------------- | ------------------------------------------- | ------------------------------ |
| **Renderer**                     | UI 渲染、用户交互、音频采集 (Web Audio API) | 与窗口同生命周期               |
| **Main**                         | IPC 路由、服务编排、文件系统、窗口管理      | 应用生命周期                   |
| **翻译服务进程**                 | CTranslate2 翻译推理                        | 常驻，按需启动，异常自动重启   |
| **whisper-server / whisper-cli** | Whisper 语音识别                            | 常驻服务优先，CLI 作为兜底路径 |

**设计原则**：

- ML 推理绝不在主进程中阻塞执行
- 渲染进程只做 UI，不直接访问任何系统资源
- 所有跨进程通信走 IPC，类型安全

## 二、核心数据流

### 2.1 键盘输入 → 翻译

```
用户输入 → tinyld 语种检测 → IPC saveMessage → SQLite 存储
                                     ↓
                              翻译开关开启?
                          ┌── 是 → 检查模型 → 可用 → Python 推理 → 更新 UI + DB
                          └── 否 → 仅显示原文
```

### 2.2 语音输入 → 流式识别 → 翻译

```
麦克风权限检查 → AudioWorklet → Ring Buffer → VAD 检测
                ↓ (检测到说话)
         增量缓冲 → 静音阈值触发分段 → 送入 Main
                ↓
     whisper-server（优先） / whisper-cli（回退）
                ↓
      文本 + 语种 → 草稿/稳定双层状态 → 实时 UI 更新
                ↓
      稳定片段进入流式翻译缓冲 → 录音结束后保存完整消息
```

### 2.3 流式翻译

```
长文本 → 按句号/问号/感叹号分句
    → 句子1 → Python 翻译 → 立即显示译文1
    → 句子2 → Python 翻译 → 追加译文2
    → ...
    → 完成 → 更新数据库
```

## 三、IPC 通信协议

所有 IPC 通道定义在 `src/shared/ipc-channels.ts`，采用命名空间式组织：

```
APP:          get-platform, get-microphone-access-status, request-microphone-access
STORAGE:      get-messages, save-message, update-translation, clear-messages,
              delete-message, export-messages
TRANSLATE:    text, text-stream, text-stream-chunk, detect-language,
              check-model, check-installed, install-model, install-pair,
              delete-pair, all-model-status, install-progress
WHISPER:      transcribe, stream-segment, check-availability,
              stream-start, stream-audio, stream-end, stream-partial, stream-commit
AUDIO:        start-recording, stop-recording, status, toggle
MODEL:        download, status, download-progress
PERFORMANCE:  get-report, take-snapshot
```

- `invoke/handle` 用于请求-响应模式
- `send/on` 用于单向事件推送（进度、流式数据）

## 四、安全设计

| 安全措施                 | 说明                                           |
| ------------------------ | ---------------------------------------------- |
| `contextIsolation: true` | 渲染进程与 Node.js 完全隔离                    |
| `nodeIntegration: false` | 渲染进程不能直接 require Node 模块             |
| contextBridge 白名单     | 仅暴露明确定义的 IPC API                       |
| Zod Schema 校验          | 关键 IPC 入参在 Main 进程统一校验              |
| Rate Limiter             | 翻译与存储写入限流，降低滥用和意外刷写风险     |
| 输入校验                 | 消息长度限制 (MAX_MESSAGE_LENGTH = 5000)       |
| Buffer 限幅              | 流式音频上传前限制最大采样数，避免内存滥用     |
| SHA256 校验              | Whisper 模型下载与翻译模型 manifest 完整性校验 |
| 无远程通信               | 所有 AI 推理本地执行，用户数据不离开设备       |
