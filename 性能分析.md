# 性能分析

> 日期：2026-04-14  
> 版本：v1.0

---

## 一、测试环境

| 项目         | 规格                                                   |
| ------------ | ------------------------------------------------------ |
| 操作系统     | macOS (Apple Silicon / Intel)                          |
| Electron     | 41.x (Chromium 124+, Node.js 20+)                      |
| Whisper 模型 | ggml-base.bin (~148MB)                                 |
| 翻译模型     | Opus-MT CTranslate2 格式 (~50-100MB/对)                |

## 二、内存占用分析

### 2.1 各阶段内存占用

| 状态                             | RSS (预估)   | 堆内存 (预估) | 说明                                 |
| -------------------------------- | ------------ | ------------- | ------------------------------------ |
| **应用空闲（未加载模型）**       | ~150-200 MB  | ~30-50 MB     | Electron + React 基础开销            |
| **Whisper 模型已下载且预热完成** | ~350-600 MB  | ~35-60 MB     | 常驻 `whisper-server` 保持模型驻留   |
| **聊天界面有历史消息**           | ~180-220 MB  | ~40-60 MB     | 100 条消息的 DOM + Zustand 状态      |
| **Whisper 推理中**               | ~500-800 MB  | ~50-70 MB     | `whisper-server` 常驻推理或 CLI 回退 |
| **翻译推理中**                   | ~400-600 MB  | ~50-70 MB     | Python 子进程 + CTranslate2 模型     |
| **同时推理**                     | ~800-1200 MB | ~60-80 MB     | Whisper + 翻译同时运行               |

### 2.2 内存分布

- **Electron 基础开销**：约 150MB（Chromium 渲染进程 + Node.js 主进程），这是 Electron 的固有代价。
- **Whisper base 模型**：~148MB。当前优先通过常驻 `whisper-server` 预热并保持模型驻留，换取更低首段识别延迟；若服务端不可用则回退到 CLI。
- **CTranslate2 翻译模型**：~50-100MB/语言对，加载到翻译服务进程中，独立于 Electron 进程。

### 2.3 内存优化建议

| #   | 方向                                              | 预期收益              | 实施难度 |
| --- | ------------------------------------------------- | --------------------- | -------- |
| 1   | 空闲一段时间后关闭常驻 Whisper / 翻译服务         | 释放 100-300MB        | 中       |
| 2   | 虚拟滚动：消息超过 500 条时用 `react-window` 替换 | 减少 DOM 内存占用     | 中       |
| 3   | Whisper 量化模型 (q5_0)                           | 模型内存占用减少 ~40% | 低       |

## 三、CPU 占用分析

### 3.1 各操作 CPU 使用率

| 操作                 | CPU 峰值 (预估) | 持续时间     | 说明                     |
| -------------------- | --------------- | ------------ | ------------------------ |
| **应用空闲**         | < 1%            | -            | 事件驱动，无轮询         |
| **Whisper 推理**     | 80-100%（多核） | 2-10s        | CPU 密集型，占满可用核心 |
| **CTranslate2 翻译** | 30-60%          | 0.5-2s       | INT8 量化后相对轻量      |
| **音频录制**         | 5-10%           | 录音期间持续 | Web Audio API 采集       |
| **VAD 语音检测**     | 2-5%            | 录音期间持续 | 实时能量分析             |

### 3.2 CPU 优化建议

| #   | 方向                      | 预期收益                                        | 实施难度 |
| --- | ------------------------- | ----------------------------------------------- | -------- |
| 1   | 调整 OMP_NUM_THREADS      | 控制 Whisper 并发线程数，平衡推理速度与系统响应 | 低       |
| 2   | Whisper 量化模型 (q5_0)   | 推理时间减少 ~30%                               | 低       |
| 3   | Metal / CUDA 加速 Whisper | 推理速度提升 3-10x                              | 高       |

> 当前 Whisper 与翻译都运行在独立进程中，均不阻塞主进程；Whisper 额外带有健康检查与自动重启机制。

## 四、GPU 分析

翻译目前仍以 CPU 为主；Whisper 在 macOS 打包构建中默认以 Metal 后端编译，运行时会优先使用可用的本地加速路径。

### 4.1 GPU 加速可行性

| 引擎        | GPU 支持              | 说明                                                 |
| ----------- | --------------------- | ---------------------------------------------------- |
| Whisper.cpp | ✅ Apple Metal / CUDA | 当前 `build-whisper.sh` 在 macOS 默认启用 Metal 编译 |
| CTranslate2 | ✅ CUDA               | 需安装 CUDA 版本的 ctranslate2                       |

### 4.2 GPU 加速预期收益

| 场景            | CPU 耗时 | GPU 耗时 (预估) | 加速比 |
| --------------- | -------- | --------------- | ------ |
| Whisper 5s 音频 | 2-4s     | 0.3-0.8s        | 3-10x  |
| 翻译单句        | 0.3-0.8s | 0.1-0.3s        | 2-3x   |

### 4.3 GPU 加速建议

- **短期**：继续沿用当前打包策略，在 macOS 构建阶段直接产出启用 Metal 的 whisper 二进制。
- **长期**：考虑为 CTranslate2 提供 CUDA 版本的安装指引，针对 Windows NVIDIA 用户。

## 五、启动时间分析

### 5.1 冷启动耗时分解

| 阶段                | 耗时 (预估) | 说明                   |
| ------------------- | ----------- | ---------------------- |
| Electron 进程启动   | ~500ms      | Chromium 启动          |
| 主进程初始化        | ~200ms      | 服务实例化             |
| SQLite 数据库初始化 | ~50ms       | 建表 + 索引            |
| Renderer 加载       | ~300ms      | Vite 构建产物加载      |
| React 首次渲染      | ~200ms      | 组件树挂载             |
| 历史消息加载        | ~100ms      | SQLite 查询 + 状态更新 |
| **总计**            | **~1.3s**   | 不含模型加载           |

### 5.2 模型首次加载耗时

| 模型                 | 首次加载耗时 (预估) | 说明                                                             |
| -------------------- | ------------------- | ---------------------------------------------------------------- |
| Whisper base         | 2-5s                | 首次下载后需要启动 `whisper-server` 并加载模型；后续可由预热摊平 |
| CTranslate2 翻译模型 | 1-3s                | 翻译服务启动 + 模型加载                                          |

### 5.3 启动时间优化建议

- **延迟模型加载**：当前已实现——模型仅在首次使用后驻留
- **Whisper 预热**：当前已实现——应用 ready 后异步预热常驻服务，减少首段识别延迟
- **预热翻译服务**：尚未实现，可在空闲时预启动翻译服务，进一步减少首次翻译延迟 1-3s
- **SQLite WAL 模式**：当前已实现，提升并发读写性能

### 6.1 翻译延迟

## 九、性能监控集成

应用内置了 `PerformanceService`，可在运行时获取性能数据：

- **IPC 调用 `performance:get-report`**：获取 JSON 格式的性能报告（内存、CPU、计时数据）
- **操作计时**：Whisper 转写和翻译操作均自动记录耗时
- **内存快照**：应用启动、初始化完成时自动采集，也支持 `performance:take-snapshot` 手动抓取

开发者可在 Electron DevTools Console 中执行：

```javascript
const report = await window.electronAPI.getPerformanceReport();
console.log(JSON.parse(report));
```

## 十、结论

本应用在桌面级硬件上的性能表现符合预期：

- **启动时间 < 2s**（首屏不含模型下载；Whisper 预热在后台异步进行）
- **翻译延迟 < 1s**（单句）
- **内存空闲 < 200MB**（未预热模型时）
- **流式语音识别**显著改善用户感知延迟
- **流式翻译**使长文本翻译体验更流畅

主要瓶颈在 **Whisper 模型驻留带来的内存占用**、**Whisper CPU / Metal 推理耗时**，以及**翻译服务首次启动**。整体架构的进程隔离与自动恢复设计确保了推理异常不会阻塞 UI 响应。
