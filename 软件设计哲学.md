# 软件设计哲学

> 项目：对话翻译桌面应用 (Dialogue Translation Application)  
> 版本：v1.0  
> 日期：2026-04-14

---

## 一、本地优先（Local-First）

**全部 AI 推理在设备本地执行**，这不仅是一个技术决策，更是一种价值观。

在 API 调用无处不在的时代，选择本地推理意味着：

- **隐私无条件保障**：用户的对话内容永远不经过任何第三方服务器。
- **网络无关**：飞机上、山洞里、无信号的地铁中——只要设备有电，应用照常工作。
- **可预测的成本**：没有 token 计费，没有 API 配额，用多少都是零边际成本。

代价是模型体积（Whisper ~148MB + 每个语言对 ~50-100MB）和推理速度不如云端 GPU。但这恰恰体现了一个核心取舍：**为用户主权付出合理的技术代价**。

当用户数据可以完全由用户自己掌控，当应用不需要因为某个服务器宕机或 API 涨价而停止工作，这种"本地优先"的设计就不只是技术上的偏好，而是对用户利益的一种保护。

---

## 二、渐进式能力加载

应用不会在启动时要求用户下载所有模型。

- Whisper 模型在首次使用语音输入时才提示下载，下载完成后再异步预热常驻识别服务
- 翻译模型在用户选择目标语言时按需检查，并通过模型管理器补齐缺失语言对
- 开发环境下 Python 依赖在首次触发翻译 / 转换时才真正成为必需；生产安装包则尽量内置运行时

这种"用到再装"的设计降低了首次体验的门槛。用户可以先当一个纯文字记录工具使用，逐步解锁语音和翻译能力。

这背后的理念是：**不要让最复杂的功能成为最简单使用场景的障碍**。一个只想打字记录的用户，不应该被迫等待 300MB 的模型下载。

---

## 三、流式体验优于批量完美

流式语音识别和流式翻译，本质上是同一个设计理念：**让用户尽早看到部分结果，而非等待完美结果**。

- **流式语音**：说话过程中逐段显示识别文字，并区分 committed / draft 两层状态，而非全部说完后一次给出。
- **流式翻译**：长文本在达到标点或长度阈值后逐块翻译、逐块展示，而非全部翻完后一次替换。

从技术上讲，分段处理可能比整体处理略损精度（上下文断裂）。但从用户体验上，即时反馈比计算最优解更重要。

这遵循了一个朴素的 HCI 原则：**系统的响应延迟感知，与实际延迟无关，与信息可见性有关**。

用户看到文字逐渐"长出来"，会觉得系统在积极响应；而看到 5 秒的空白等待后突然出现一大段文字，即使总耗时更短，主观感受也更差。

---

## 四、进程隔离即韧性

应用刻意将 ML 推理放在独立进程中：

- **Whisper** → 常驻 `whisper-server` 优先，带健康检查与自动重启；必要时回退到 CLI
- **翻译** → 常驻翻译服务进程，异常后指数退避重启
- **UI** → Chromium 渲染进程（与计算完全解耦）

如果 Whisper 服务意外退出，主进程会先尝试自动恢复；若恢复失败，仍可以回退到 CLI 或向 UI 报错，而不是让整个桌面应用闪退。

这种设计比把所有计算塞进同一个进程更健壮，尽管多了进程间通信的复杂度。代价是明确的（IPC 序列化开销），收益也是明确的（任何一个推理引擎的故障都不会导致用户丢失数据或被迫重启应用）。

**系统的鲁棒性，不在于单个组件有多可靠，而在于任何组件失败时，其他组件能否继续正常工作。**

---

## 五、简单胜于万能

这个应用有意限制了功能边界：

- 只做对话记录 + 翻译，不做实时会议转写
- 只支持 9 种语言，不追求覆盖 200+ 语言
- 只做文本翻译，不做文档翻译
- 不做用户账号系统、云同步、协作功能

每个"不做"都是为了让"做了的"部分足够简单可靠。

功能膨胀是软件腐化的常见路径。一旦开始添加"顺便也做一下"的功能，每个功能都会带来维护负担、测试表面积和用户认知成本。当一个工具试图解决所有问题时，它往往哪个问题都解决不好。

**一个小而完整的桌面工具，比一个功能繁多但半成品的平台更有价值。**

---

## 六、总结

这五条原则并不独立，而是相互支撑的：

- **本地优先**决定了技术边界（不能依赖云端）；
- **渐进式加载**让本地模型的体积代价变得可接受，同时保留安装包层面的开箱即用；
- **流式体验**弥补了本地推理速度不如云端的感知差距；
- **进程隔离**与自动恢复为本地多引擎集成提供了安全网；
- **简单原则**确保整个系统不会因为功能蔓延而失控。

最终的产品应该是：**一个打开就能用、说话就能译、关掉不留痕的桌面工具**。不多，不少。
